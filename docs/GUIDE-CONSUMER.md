# CKM Consumer Guide

How to add `ckm [topic]` to your CLI tool.

---

## Who is this for?

You have a CLI tool (like VersionGuard) and a `ckm.json` file (generated by forge-ts or another producer). You want your users to be able to type `mytool ckm calver` and get structured help.

## Prerequisites

- A `ckm.json` file in your project (usually at `docs/ckm.json`)
- Node.js 18+ (for the npm package)

## Install

```bash
npm install ckm-sdk
```

## Integration (3 steps)

### Step 1: Load the manifest

```javascript
const { createCkmEngine } = require('ckm-sdk');
const fs = require('fs');

const manifest = JSON.parse(fs.readFileSync('docs/ckm.json', 'utf-8'));
const engine = createCkmEngine(manifest);
```

The engine accepts both v1 and v2 manifests. v1 is auto-migrated internally.

### Step 2: Wire into your CLI

**Commander.js:**

```javascript
const { Command } = require('commander');
const program = new Command();

program
  .command('ckm [topic]')
  .description('Codebase Knowledge Manifest — browse docs and help')
  .option('--json', 'Machine-readable output for LLM agents')
  .action((topic, { json }) => {
    if (json) {
      console.log(JSON.stringify(engine.getTopicJson(topic), null, 2));
    } else if (topic) {
      const content = engine.getTopicContent(topic);
      if (!content) {
        console.error(`Unknown topic: ${topic}`);
        console.log(engine.getTopicIndex('mytool'));
        process.exit(1);
      }
      console.log(content);
    } else {
      console.log(engine.getTopicIndex('mytool'));
    }
  });
```

**That's it.** Your users now have:

```bash
mytool ckm                    # List all topics
mytool ckm calver             # Deep dive on CalVer
mytool ckm calver --json      # Structured JSON for LLM agents
mytool ckm --json             # Full index as JSON
```

### Step 3: Embed the manifest at build time (optional)

If you use a bundler (Vite, esbuild, etc.), embed the manifest at build time so there's no runtime file read:

```typescript
// Vite raw import — embedded at build time
import ckmRaw from '../docs/ckm.json?raw';
import { createCkmEngine } from 'ckm-sdk';

const engine = createCkmEngine(JSON.parse(ckmRaw));
```

This is what VersionGuard does. The `ckm.json` becomes part of the binary — no external file needed at runtime.

## Engine API

### `engine.getTopicIndex(toolName)`

Returns a formatted topic list for terminal display.

```
mytool CKM — Codebase Knowledge Manifest

Usage: mytool ckm [topic] [--json] [--llm]

Topics:
  calver              Configures CalVer validation rules.
  semver              Configures SemVer validation rules.
  git                 Configures git-related enforcement.

Flags:
  --json    Machine-readable CKM output (concepts, operations, config schema)
  --llm     Full API context for LLM agents (forge-ts llms.txt)
```

### `engine.getTopicContent(topicName)`

Returns formatted detail for a single topic. Returns `null` if not found.

```
# Configures CalVer validation rules.

## Concepts

  CalVerConfig — Configures CalVer validation rules.
    format: CalVerFormat = YYYY.MM.DD
      Calendar format used for version strings.

## Operations

  validate() — Validates a CalVer string.
    @param version: Version string to validate.

## Config Fields

  calver.format: CalVerFormat = YYYY.MM.DD
    Calendar format used for version strings.

## Constraints

  [constraint-no-future-dates] CalVer versions must not reference future dates.
    Enforced by: validate
```

### `engine.getTopicJson(topicName)`

Returns structured data. Behavior depends on the argument:

- **No argument:** Returns the full topic index with counts (Level 2)
- **Valid topic:** Returns the topic's concepts, operations, config, constraints (Level 1J)
- **Unknown topic:** Returns `{ error: "Unknown topic: foo", topics: ["calver", "semver", ...] }`

### `engine.inspect()`

Returns manifest statistics:

```javascript
{
  meta: { project: "my-tool", language: "typescript", generator: "forge-ts@1.1.0", generated: "..." },
  counts: { concepts: 24, operations: 23, constraints: 0, workflows: 0, configKeys: 44, topics: 13 },
  topicNames: ["calver", "semver", "git", ...]
}
```

### `engine.getManifest()`

Returns the raw v2 manifest (auto-migrated if input was v1).

### `engine.topicsCount`

Number of derived topics (property, not method).

## Other SDK Functions

### `validateManifest(data)`

Validate any manifest against the v2 schema:

```javascript
const { validateManifest } = require('ckm-sdk');
const result = validateManifest(manifest);
// { valid: true, errors: [] }
// or { valid: false, errors: [{ path: "/concepts/0/slug", message: "Missing required field" }] }
```

### `detectVersion(data)`

Returns `1` or `2`.

### `migrateV1toV2(data)`

Converts a v1 manifest to v2 format. Returns the migrated manifest.

## Rust Consumer

If your CLI is written in Rust:

```bash
cargo add ckm
```

```rust
use ckm::CkmEngine;

let data: serde_json::Value = serde_json::from_str(&manifest_json).unwrap();
let engine = CkmEngine::new(data);

println!("{}", engine.topic_index("mytool"));

if let Some(content) = engine.topic_content("calver") {
    println!("{}", content);
}
```

Same API, same behavior. The Rust crate IS the engine — Node.js calls into it via napi-rs.

## Standalone CLI

If you just want to browse a `ckm.json` without writing code:

```bash
# Via npm (CLI is built into ckm-sdk)
npx ckm-sdk browse --file docs/ckm.json
npx ckm-sdk browse calver --file docs/ckm.json
npx ckm-sdk browse --json --file docs/ckm.json
npx ckm-sdk validate docs/ckm.json
npx ckm-sdk inspect docs/ckm.json

# Or via Rust native binary
cargo install ckm-cli
ckm browse --file docs/ckm.json
```

## Progressive Disclosure Protocol

CKM enforces four disclosure levels with token budgets:

| Level | What the user types | What they get | Token budget |
|-------|-------------------|---------------|-------------|
| 0 | `mytool ckm` | Topic list | 300 |
| 1 | `mytool ckm calver` | Topic detail | 800 |
| 1J | `mytool ckm calver --json` | Structured JSON | 1200 |
| 2 | `mytool ckm --json` | Full index JSON | 3000 |

LLM agents use `--json` to get structured data under tight token budgets. Humans get formatted text. Same command, same data, different presentation.

## Real-World Example: VersionGuard

VersionGuard's complete CKM integration is 3 lines + the command handler:

```typescript
// cli.ts
import ckmRaw from '../docs/ckm.json?raw';
import { createCkmEngine } from 'ckm-sdk';

const engine = createCkmEngine(JSON.parse(ckmRaw));

program
  .command('ckm [topic]')
  .option('--json', 'Machine-readable CKM output')
  .action((topic, { json }) => {
    if (json) {
      console.log(JSON.stringify(engine.getTopicJson(topic), null, 2));
    } else if (topic) {
      const content = engine.getTopicContent(topic);
      if (!content) {
        console.error(`Unknown topic: ${topic}`);
        console.log(engine.getTopicIndex('vg'));
        process.exit(1);
      }
      console.log(content);
    } else {
      console.log(engine.getTopicIndex('vg'));
    }
  });
```

That replaced 690 lines of handrolled CKM code with a 3-line import + standard command handler. Build passes, all 279 tests pass.
